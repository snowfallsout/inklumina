# DEVNOTES_2026-05-02T130000Z

## 背景

在第一輪修復完成後，`npm run check` 已從大量 import/path 錯誤收斂到 mobile flow 的 Svelte 5 警告。這些問題不再是基礎結構缺口，而是元件邊界尚未對齊 Svelte 5 基線：

- 子元件仍使用 `createEventDispatcher` 與 `on:` custom event 模式。
- 會驅動畫面的 page-local 值仍以普通 `let` 存放，而不是 `$state(...)`。
- `MBTIMobilePage.svelte` 仍保留從 `static/mobile.html` 搬來、但目前 DOM 已不再使用的 CSS 區段。
- `MBTICard.svelte` 先前把純展示字元做成可點擊節點，造成不必要的 a11y 與互動語義偏差。

因此這一輪目標不是再擴大重構，而是把 mobile 組件群收斂到一個乾淨、可繼續演進的 Svelte 5 slice。

## 本次修改

### 1. Mobile component contract 改為 callback props

以下元件已從 `createEventDispatcher` 改為 callback props：

- `src/lib/components/mobile/DimButton.svelte`
- `src/lib/components/mobile/DimRow.svelte`
- `src/lib/components/mobile/SubmitButton.svelte`
- `src/lib/components/mobile/HoloCardPreview.svelte`

調整後的方向：

- `DimButton` 只負責回傳 `ontoggle`。
- `DimRow` 負責把 child callback 收斂成 `onselect({ dimension, value })`。
- `SubmitButton` 只接受 `onsubmit`。
- `HoloCardPreview` 在產圖後只透過 `ongenerated` 回傳結果，不再走舊 event dispatcher。

這使 mobile flow 的資料回傳路徑與 Svelte 5 基線一致，也避免繼續擴寫舊 component event 模式。

### 2. Page-local state 改為 rune state

`src/lib/components/mobile/MBTIMobilePage.svelte` 已將以下局部狀態改為 `$state(...)`：

- `sel`
- `loading`
- `previewRef`
- `screen`

同時補上局部型別：

- `MobileScreen`
- `LuckyColorPayload`
- `DimRowSelection`
- `HoloCardPreviewHandle`

目的很直接：讓 UI 切換、socket 回應、產圖指令與選擇陣列的更新都落在 Svelte 5 明確可追蹤的 reactive 模型內，而不是依賴舊有 top-level `let` 的隱性行為。

### 3. MBTI card 退回純展示語義

`src/lib/components/mobile/MBTICard.svelte` 已移除不被 page 使用的點擊選字邏輯，保留為純展示卡片：

- 不再 dispatch `select`
- 不再讓字母格承擔互動責任
- 補上 keyed each，避免不必要的 DOM diff 模糊點

這符合目前 page composition 的真實資料流：真正的選擇入口在 `DimRow`，卡片只反映目前選擇結果。

### 4. 移除已失效的 copied CSS

`src/lib/components/mobile/MBTIMobilePage.svelte` 已刪除大段不再對應現有 DOM 的 CSS：

- 舊 welcome / mbti / result id selector
- 已拆進 child component 的 card / dim button 樣式
- 不再被使用的 hidden-state selector

保留的只剩 page-level 背景、按鈕、標題與 layout 樣式。這讓 Svelte warning 清零，也避免之後調整子元件時被 page 端的殘留樣式反向污染。

## 影響範圍

這一輪只收斂 mobile slice，不處理以下區域：

- display footer / `JoinQr.svelte`
- `static/**` legacy source
- socket/server 契約更完整的共享型別抽離

換句話說，現在 mobile flow 已經先達到可繼續工作的 Svelte 5 基線；下一輪若要繼續，應該把注意力轉向跨端 socket contract 與 display state ownership，而不是再回頭補 mobile 的基礎語法債。
