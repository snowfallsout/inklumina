# DEVNOTES_2026-05-02T212000Z

## 背景

在先前幾輪重構之後，`src/lib/function/runtime/` 已經只剩五個純 kernel 檔案：

- `core.ts`
- `draw.ts`
- `gesture.ts`
- `particle.ts`
- `sprite.ts`

它們已不再承擔 runtime owner / service orchestration 職責，只被 `services/display/*` 取用。

因此，繼續保留 `function/runtime` 這個路徑會產生兩個誤導：

1. 名稱仍像是舊 runtime owner
2. `function/` 看起來仍是 active display 架構的一級目錄

## 本輪決策

### A. 將 `function/runtime` 重新命名為 `services/display/kernel`

本輪把剩餘 kernel 檔案移到：

- `src/lib/services/display/kernel/*`

讓路徑直接反映其角色：display service 轄下的 kernel。

### B. `function/` 不再承載 active code

本輪同步移除 `src/lib/function/_CORE_FILES_.md`，改為在 kernel 目錄中放置新的說明檔，讓 `function/` 不再是 active code 或 active documentation 的入口。

### C. 更新相鄰說明與 import

所有原本指向 `$lib/function/runtime/core` 的 service import 改為：

- `$lib/services/display/kernel/core`

並修正仍提到 `function/*` 的相鄰說明文字。

## 本輪不做

本輪刻意不做以下事項：

- 不再變更 kernel 內部行為
- 不重寫 `services/display/*` API
- 不調整 `static/*` reference baseline

## 驗證方式

本輪完成後需確認：

- `npm run check` 通過
- workspace 中不再有 `function/runtime` 或 `$lib/function/` 的 active import
- `src/lib/function/` 可被視為已退出 active code 結構

## 下一步

若本輪穩定，下一步應優先考慮：

1. 是否要把 `services/display/kernel/*` 再拆成更細的 `particle` / `render` / `gesture` 子層
2. 是否要為 `kernel/core.ts` 與 `services/display/*` 之間補一份 boundary 說明
3. 是否要把 `services/display/kernel/README.md` 精簡成只保留 active boundary 與禁止事項。
