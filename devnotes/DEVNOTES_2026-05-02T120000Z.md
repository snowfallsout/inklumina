# DEVNOTES_2026-05-02T120000Z

## 背景

目前專案的 `src/` 已經進入 Svelte 5 遷移中段，但實際狀態不是單純「語法還沒改完」，而是同時存在以下三種狀況：

- 新的 Svelte 5 state/module 結構已經出現，例如 `src/lib/state/*.svelte.ts`。
- 舊的 namespace 與舊的 DOM 驅動模型仍殘留在 `src/lib/function/**`、`src/lib/components/**`、`src/app.d.ts`。
- `static/display.html`、`static/mobile.html`、`static/server.js` 仍是最完整且最穩定的原始功能來源，但依保護政策不可直接作為重構目標檔案修改。

因此，後續工作不能只以「能不能編過」為標準，而需要一份更嚴格的 Svelte 5 規範基線，清楚切分：

- 哪些屬於立即必修，否則會持續阻斷遷移。
- 哪些可以暫時保留，但必須被隔離在明確邊界內。
- 哪些只能保留作 reference，不應再成為 `src/` 的直接依賴來源。

## 參考來源

這份基線以以下來源共同判定：

1. Svelte 5 官方遷移規則。
2. 本 repo 的 `copilot-instructions.md` 與 `AGENTS.md`。
3. 現況的 `npm run check` 診斷。
4. 原始功能來源：`static/display.html`、`static/mobile.html`、`static/server.js`。

## 功能對照基線

為避免重構時偏離原始行為，先定義三份 legacy 檔案在新結構中的角色：

### 1. `static/mobile.html` 對應的功能切片

- Welcome -> MBTI 選擇 -> 提交 -> 結果卡片，這是 mobile flow 的完整來源。
- 視覺卡片漸層、四維選擇按鈕、送出 loading、Socket 回傳 lucky color、產圖與保存提示，都是功能需求，不是可任意刪減的裝飾。
- 新的 `src/lib/components/mobile/**` 可以重組結構，但不可在沒有替代設計與驗證前刪掉上述功能。

### 2. `static/display.html` 對應的功能切片

- Display canvas、camera feed、hand/face tracking、emotion badge、legend、toast、session panel、corner QR、waiting 狀態、session reset，都是 display route 的需求來源。
- 粒子引擎與手勢畫布邏輯可以保留 imperative 寫法，但 UI overlay、狀態切換、Socket 同步、session 呈現不應再維持散落的 DOM 操作。
- `static/display.html` 只作行為比對，不再作為 `src` 的直接 import 或 path 依賴來源。

### 3. `static/server.js` 對應的功能切片

- Session REST API、socket `state`、`submit_mbti`、`spawn_particles`、`session_reset` 是 server 契約來源。
- `src/lib/server/**` 與 `src/lib/services/socket.ts` 應以此為事件契約基礎，建立型別清楚的 client/server 對接。
- server 端資料結構可以重寫，但事件名稱、payload 欄位與使用時序不能任意漂移。

## 嚴格 Svelte 5 規範基線

以下分成三級：A 級必改、B 級可暫留、C 級只可保留作參考。

## A 級：一定要重構，否則不算完成遷移

### A-1. 不存在的 namespace 或錯誤路徑不得存在於 `src/`

判準：

- `src/` 內禁止再出現對不存在模組的 import。
- 特別是 `$lib/runes/*`、`$lib/features/display/*`、`$lib/function/coordinator` 這類過渡期或錯誤路徑，必須改到目前真實存在的檔案結構。

原因：

- 這不只是風格問題，而是直接的 build blocker。
- 只要 namespace 不收斂，後續型別整理與功能驗證都會失真。

當前判定：

- 屬於立即必修。
- 相關問題已出現在 `src/lib/components/display/**`、`src/lib/services/**`、`src/lib/state/**`、`src/routes/display/+page.svelte`、`src/lib/function/runtime/**`。

### A-2. 不可用 `src/app.d.ts` 假宣告不存在的 rune 模組來遮掩遷移問題

判準：

- `app.d.ts` 只能補 framework 正常需要的型別擴充，不可用來假裝不存在的 `$lib/runes/*` 模組合法。
- 不可用大面積 `any` 宣告來掩蓋真實型別或錯誤 import。

原因：

- 這會讓 typecheck 表面通過，但實際結構錯誤被隱藏。
- 任何依賴 fake declaration 的修復都會讓後續重構成本更高。

當前判定：

- 屬於立即必修。

### A-3. Svelte 5 runes mode 元件中，凡會驅動畫面更新的局部狀態，必須使用 `$state`

判準：

- 在 `.svelte` 中，若變數被重新賦值且會影響 render，就不能保留成普通 `let`。
- `loading`、`screen`、`imageUrl`、`previewRef` 這類會影響 UI 的值，必須明確改成 `$state(...)` 或移到更合適的 state owner。

原因：

- 這是 Svelte 5 官方 migration guide 的直接要求。
- 若保留普通 `let`，更新可能不會正確觸發，形成假性成功。

當前判定：

- `src/lib/components/mobile/HoloCardPreview.svelte`
- `src/lib/components/mobile/MBTIMobilePage.svelte`
- 以上屬立即必修。

### A-4. 新增或維護中的 Svelte 5 元件，不再接受 Svelte 4 component event 模式作為目標寫法

判準：

- `createEventDispatcher` 不再作為新基線。
- 元件對父層回傳資料時，應優先改用 callback props。
- 元件 DOM 事件改用 `onclick` 等事件屬性，而非 `on:` 指令，除非有明確兼容性理由。

原因：

- Svelte 5 官方已將 `createEventDispatcher` 視為 deprecated 路徑。
- 若 mobile flow 繼續以舊事件模型擴寫，後續整體風格會持續分裂。

當前判定：

- `src/lib/components/mobile/DimButton.svelte`
- `src/lib/components/mobile/DimRow.svelte`
- `src/lib/components/mobile/SubmitButton.svelte`
- `src/lib/components/mobile/HoloCardPreview.svelte`
- `src/lib/components/mobile/MBTICard.svelte`
- `src/lib/components/mobile/MBTIMobilePage.svelte`
- 以上都應列入必修，而不是單純「之後再說」。

### A-5. `.svelte.ts` 共享狀態模組不可同時成為主要 DOM owner

判準：

- `.svelte.ts` 可以持有共享 reactivity 與封裝邏輯。
- 但若模組直接負責 `document.createElement`、`document.body.appendChild`、控制特定 DOM 節點生命週期，就已超出狀態模組職責。
- 畫面節點的建立、綁定、銷毀，應優先回到 component 或明確的 browser adapter。

原因：

- 共享 state 與 DOM ownership 混在一起時，SSR 邊界、測試能力、生命週期管理都會失控。
- 這正是目前 smile/media 狀態模組最容易持續惡化的地方。

當前判定：

- `src/lib/state/smile.svelte.ts`
- `src/lib/state/media.svelte.ts`
- 兩者都應重構，不應視為可永久保留。

### A-6. 與原始 server 契約有關的 socket/session 結構必須先型別化，再繼續擴充功能

判準：

- `state`、`spawn_particles`、`session_reset`、`submit_mbti`、`lucky_color` 等事件，需要明確 type/interface。
- client service 與 server 事件 payload 要對齊 `static/server.js` 的欄位與時序。

原因：

- 現在大量 `any` 讓 display/mobile/server 三端契約漂移風險很高。
- 只要事件 payload 失配，畫面表面仍可能渲染，但邏輯會悄悄偏移。

當前判定：

- `src/lib/services/socket.ts`
- `src/lib/state/session.svelte.ts`
- `src/lib/server/**`
- 這一塊屬於遷移核心，不可延後到最後。

## B 級：可以暫留，但必須被隔離並標記為過渡層

### B-1. 粒子引擎與畫布演算法可以暫時維持 imperative TS，不強制全面 rune 化

允許條件：

- 核心演算法可保留在普通 `.ts` 檔。
- 不直接依賴不存在的 namespace。
- 不把 DOM 查找、Svelte UI 控制、socket 綁定混在同一層。
- 有明確 entry owner，例如 `Canvas.svelte` 或單一 coordinator 負責接線。

原因：

- `display.html` 原始邏輯本來就是高度 imperative 的 canvas engine。
- 這一塊的重點是邊界清楚，不是為了「看起來像 Svelte」而硬改成 rune 寫法。

當前判定：

- `src/lib/core/**`
- `src/lib/utils/particle.ts`
- `src/lib/utils/pool.ts`
- `src/lib/function/runtime/**`
- 可以暫留，但前提是先解除錯誤 import 與錯誤耦合。

### B-2. 與瀏覽器 API 高耦合的 camera / mediapipe adapter 可以先留在 client-only service

允許條件：

- 必須明確包在 browser guard 內。
- 不在 module evaluation 時直接操作 `window` 或 `document`。
- 對外只暴露型別清楚的 API 與資料結構。

原因：

- 這些模組天然依賴 camera、MediaPipe、DOM、rAF，不適合純 state 模組化。
- 但它們必須退回 adapter/service 邊界，而不是散落在 component 和 state 之間。

### B-3. 某些 `any` 可作為暫時過渡，但只允許留在外部庫接縫處

允許條件：

- 僅限第三方未型別化輸入，例如 MediaPipe raw result、window 掛載的外部 constructor。
- 必須逐步被 `unknown` + narrowing 或 interface 替代。
- 禁止把 `any` 擴散到共享 state、component props、socket payload。

原因：

- 真正難型別化的地方通常是外部 runtime 邊界，不是內部領域模型。

## C 級：只可保留作參考，不可繼續作為新架構依賴目標

### C-1. `static/display.html`、`static/mobile.html`、`static/server.js` 是 reference，不是直接改造目標

判準：

- 僅作功能來源、視覺比對、事件契約比對。
- 不再讓 `src/` 對它們形成 runtime 依賴。
- 不以複製貼上方式持續同步新邏輯回 `static/`。

原因：

- `static/**` 依 repo policy 屬 protected path。
- 同時維護兩套實作會讓遷移永遠停留在中間態。

### C-2. `src/lib/function/**` 內的 legacy wiring 只能作過渡參考，不應再擴張

判準：

- 若某個功能尚未搬完，可短期讀取其邏輯。
- 但新的功能不得繼續往這一層追加。
- 新增邏輯應優先放入 `state`、`services`、`utils`、`components` 中對應的最小責任位置。

原因：

- 若 migration 期間持續加碼 legacy runtime，之後只會出現更多雙軌結構。

## 一步步的後續重構順序

以下順序是這份基線推薦的實際執行順序：

1. 先修所有錯誤 import 與錯誤 namespace，讓 `src/` 至少對齊真實檔案結構。
2. 拿掉 `app.d.ts` 中用來遮掩錯誤路徑與過寬 `any` 的假宣告。
3. 修 mobile flow，先把 `createEventDispatcher`、`on:` custom events、非 reactive `let` 收斂成 Svelte 5 合法基線。
4. 修 display state ownership，把 `smile.svelte.ts`、`media.svelte.ts` 中的 DOM ownership 拆回 component 或 adapter。
5. 修 socket/session 契約型別，對齊 `static/server.js` 的 payload。
6. 最後再整理粒子與 camera runtime 的 imperative 層，把可暫留的 legacy 算法隔離在乾淨邊界內。

## 本次結論

這份基線的核心原則只有一句話：

- 允許保留 imperative 演算法。
- 不允許保留錯誤邊界、錯誤路徑、假型別、假反應式。

換句話說，這個專案接下來不是要把所有東西都寫成 rune，而是要把「哪些應該是 Svelte 5 owner、哪些只是 browser runtime adapter、哪些只是 legacy reference」切乾淨。

## 驗證與使用方式

1. 之後每次開始修一塊功能前，先對照本文件，判斷該問題屬於 A、B、C 哪一級。
2. A 級項目不能再以 workaround、型別遮罩、或新增 alias 的方式延後。
3. B 級項目可以暫留，但修補時必須同時補上邊界說明與 owner。
4. C 級項目不得再擴寫，只能作行為比對。
