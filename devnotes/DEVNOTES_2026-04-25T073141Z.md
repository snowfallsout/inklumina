## 官方基準摘要

### 1. `@dev.svelte/mcp` / 官方 MCP 基線

- 官方 `cli/mcp` 文件指出，標準安裝方式是 `npx sv add mcp`。
- 官方說明指出，這個命令會產生：
  - local 或 remote MCP 設定
  - 給 agent 使用的 README / agent guidance
- 目前 repo 內沒有看到 `sv add mcp` 典型生成的 MCP 設定檔；目前只有人工整理文件，例如 `src/lib/runes/store.md` 提到 MCP/autofixer 檢查結果。

### 2. Svelte 5 shared state 基線

- 官方文件明確指出 `.svelte.ts` / `.svelte.js` 是 Svelte 5 用來承載 shared reactive logic 的標準位置。
- 官方文件同時指出：若要跨模組共享 state，建議使用可安全追蹤的匯出方式，並避免以脆弱的多層相對路徑分散引用。
- `kit/$lib` 與 project structure 文件的方向很清楚：共享模組應集中於 `src/lib`，並以 `$lib/...` 或穩定路徑引用，而不是在 UI 深層目錄裡大量使用容易算錯層級的 `../../../...`。
- `$effect` 官方定位是 browser-side side effect / DOM / canvas / 第三方整合的 escape hatch，適合 `Canvas.svelte` 這類顯示層，但不適合拿來補償整體架構上的 module resolution 問題。

## 本次核對結論

- `svelte.config.js` 已強制專案採用 runes mode。
- `src/lib/runes/` 內已有 `*.svelte.ts` 模組，代表專案已開始以 Svelte 5 shared state 形式重構。

### 尚未成立的關鍵條件

- 顯示層 `display` 仍存在大量錯誤的相對路徑 import，這不是完整 Svelte 5 架構應有的狀態。
- display runtime 仍混用新 runes、compat layer、legacy function runtime 與舊 JS service，導致編譯檢查與實際載入鏈都不穩定。
- repo 目前沒有看到官方 `sv add mcp` 生成的 MCP 設定資產，因此「已使用 Svelte MCP」和「repo 已正規化整備 MCP」是兩件不同的事；目前後者尚未完成。

## 驗證後的主要問題分類

### A. display 元件的 import 層級普遍錯誤

這是目前最直接、最確定的阻塞點。

已驗證的例子：

- `src/lib/components/display/Canvas.svelte`
  - 目前引用：`../../../core/particleEngine`
  - 但檔案位於 `src/lib/components/display`，往上三層會跳到 `src/`，不是 `src/lib/`
  - 同檔案內 `../../../runes/...`、`../../../services/...`、`../../../function/...` 也同樣多跳了一層
- `src/lib/components/display/Footer.svelte`
  - `../../../runes/mbti.svelte`
  - `../../../runes/session.svelte`
  - `../../../runes/ui.svelte`
- `src/lib/components/display/HandBadge.svelte`
- `src/lib/components/display/Toast.svelte`
- `src/lib/components/display/EmotionBadge.svelte`
- `src/lib/components/display/SessionPanel.svelte`
- `src/lib/components/display/CamToggle.svelte`
- `src/lib/components/display/Legend.svelte`
- `src/lib/components/display/QrCode.svelte`

這類錯誤表示目前 display 區域沒有遵循穩定的 `$lib` import 策略，也尚未完成路徑收斂。

### B. `QrCode.svelte` / `qrCode.svelte` 的大小寫與結構異常

`npm run check` 與單檔 diagnostics 顯示：

- `Footer.svelte` 匯入 `./QrCode.svelte` 時，TypeScript 報出與 `qrCode.svelte` 大小寫衝突
- 同時報出 module has no default export

而目前 `src/lib/components/display/QrCode.svelte` 的內容可見明顯結構不完整現象：

- 有孤立的 `prompt('Copy this URL', url);` 片段
- 缺少完整函式包裹
- 標記與樣式區塊明顯被截斷 / 拼接

這代表 QR component 本身目前不是穩定可維護狀態，即使 import path 修正，這一塊也仍需單獨整修。

### C. `src/lib/core/particleEngine.ts` 目前處於不一致狀態

本次讀檔可直接確認這個檔案內目前存在兩段 `export default class ParticleEngine`，也就是兩份實作被串接在同一檔案中。

`npm run check` 已報出：

- `A module cannot have multiple default exports`

這表示目前的 particle engine compat layer 還沒有收斂成單一 source of truth。這一點直接影響 display 畫布層是否能作為 Svelte 5 runtime 的穩定入口。

### D. `function-display-legacy` 仍大量污染 repo-wide check

`src/lib/function/function-display-legacy/` 目前仍引用多個不存在的舊模組，例如：

- `$lib/shared/contracts`
- `$lib/shared/socket-client`
- `$lib/features/display/state`
- `$lib/features/display/session`
- `$lib/core/spriteCache`

此外，`src/lib/function/types/ambient.d.ts` 內有：

- `declare module '$lib/*';`

這種過寬的 ambient declaration 會讓不存在的模組在部分檢查場景下看似可 import，但無法提供真實 exported member，因此最後在 repo-wide check 仍出現大量 `has no exported member` 類型錯誤。

這不是完整架構，而是遷移中的暫時遮罩。

### E. 舊 JS service 仍在 strict `checkJs` 下成為主要錯誤來源

`src/lib/services/camAndMedia.js` 目前仍被 `tsconfig.json` 的 `allowJs + checkJs` 納入檢查。

已驗證錯誤類型包括：

- `Cannot find name 'Hands'`
- `Cannot find name 'FaceMesh'`
- 多個 `possibly null`
- 多個 `implicitly has any`

這說明專案雖然宣告為 Svelte 5 + TypeScript，但實際上仍保留一段高噪音的 legacy JS 執行鏈，尚未完成隔離或遷移。

### F. mobile 區域仍殘留 Svelte 4/過渡期模式

`npm run check` 還顯示 `src/lib/components/mobile/MBTIMobilePage.svelte` 等檔案存在：

- `on:click` deprecated warning
- 多個 `non_reactive_update` warning

這表示 repo 不只 display 區域，mobile 區域也尚未完全統一到 Svelte 5 runes 慣例。

因此目前不能把整個 repo 判定為「Svelte 5 fully-complete architecture」。更精準的說法是：

- 基礎框架版本已是 Svelte 5
- 但應用層仍處於 migration-in-progress

## 對 `src/lib/runes/*.svelte.ts` 的判定

### 正面觀察

- `src/lib/runes/media.svelte.ts`
- `src/lib/runes/mbti.svelte.ts`
- `src/lib/runes/session.svelte.ts`

這些檔案目前至少表示專案已把 shared state 收攏到 `.svelte.ts` 模組，方向符合 Svelte 5。

### 需要注意的架構風險

官方 `.svelte.ts` 文檔對 shared state export 有明確限制與建議；目前 repo 的 rune modules 雖然可被編譯器接受，但仍建議後續再做一次規則型檢查，確認：

- 是否存在直接 export 並頻繁 reassignment 的 primitive state
- 是否應改成 object state / getter function / action API 形式
- 是否所有外部引用都已統一到同一種 import 慣例

本次檢查沒有把 `src/lib/runes/*.svelte.ts` 判定為當前主阻塞；當前主阻塞仍是 display import 鏈、QR component 破損、particleEngine 雙重實作，以及 legacy runtime 污染。

## 本次檢查結論

### 判定

- 這個 repo 是 Svelte 5 / SvelteKit 2 專案，這點成立。
- 這個 repo 已開始使用 Svelte 5 runes 與 `.svelte.ts`，這點成立。
- 這個 repo 目前尚未完成「完整、乾淨、統一」的 Svelte 5 架構，這點也成立。

### 最主要阻塞排序

1. display 元件大量錯誤相對路徑 import
2. `QrCode.svelte` / `qrCode.svelte` 結構與 casing 問題
3. `src/lib/core/particleEngine.ts` 有雙重 default export / 雙實作拼接
4. `function-display-legacy` 仍引用已不存在的舊架構模組
5. `camAndMedia.js` 等舊 JS service 在 `checkJs` 下持續產生高噪音錯誤
6. mobile 區域仍殘留 Svelte 4 過渡寫法與 non-reactive warnings

## 建議後續順序

1. 先把 display 全區 import 收斂到 `$lib/...` 或正確的 `../../...`，不要再用目前的 `../../../...`。
2. 單獨修復 `QrCode.svelte`，並統一檔名大小寫，只保留一個 canonical component。
3. 清理 `src/lib/core/particleEngine.ts`，只保留一份相容層或一份正式實作。
4. 決定 `function-display-legacy` 的策略：
   - 要嘛完全隔離出 repo-wide check
   - 要嘛補齊缺失的 shared contract / socket / state 對應
5. 對 `src/lib/runes/*.svelte.ts` 做一次針對官方 `.svelte.ts` export 規則的專項審核。
6. 維護者若同意，再把官方 `sv add mcp` 產生的 MCP 設定納入 repo。
