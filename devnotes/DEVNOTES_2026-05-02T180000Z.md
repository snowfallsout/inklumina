# DEVNOTES_2026-05-02T180000Z

## 背景

在 `services/display/*` 與 `shared/constants/*` 完成收斂之後，下一個最明確的結構殘留是：

- display state 仍由 `utils/state.ts` 持有
- `services/display/session.ts`、`function/runtime/*` 等 display service / runtime consumer 仍直接引用這個 utils 路徑

這與目前的架構共識不符，因為 `state` 層已經被定義為 Svelte 5 runes / state owner，而 `utils` 只應保留真正 generic 的 helper。

## 本輪決策

### A. `state/display.svelte.ts` 作為正式 owner

本輪新增 `state/display.svelte.ts`，承接既有 display state store 內容與 API，作為新的正式 owner。

本輪刻意不改寫 displayState 的行為模型，只先完成 owner 遷移：

- 讓 state 位置正確
- 保持既有 method API 穩定
- 不在同一輪把它重寫成不同的 public interface

### B. `utils/state.ts` 退回 compatibility shell

`utils/state.ts` 不再持有實作，只保留對 `state/display.svelte.ts` 的 re-export。

這樣可以在不破壞既有 import path 的前提下，先完成 ownership 修正。

### C. 直接 consumer 先改吃新路徑

本輪先把最核心的直接 consumer 改到新路徑：

- `services/display/session.ts`
- `function/runtime/dom.ts`
- `function/runtime/legacy.ts`
- `function/runtime/realtime.ts`
- `function/session/client.ts`

## 本輪不做

本輪刻意不做以下事項：

- 不重寫 `displayState` 成全新的 rune-only public API
- 不搬移 `utils/types.ts`
- 不處理 `function/runtime/*` 的實作位置
- 不清理 `function/session/*` compatibility shell

## 驗證方式

本輪完成後需確認：

- `npm run check` 通過
- `state/display.svelte.ts` 已成為 display state 的正式 owner
- `utils/state.ts` 僅保留 re-export，不再持有實作
- 主要 display consumer 已從 `utils/state.ts` 改為 `state/display.svelte.ts`

## 下一步

若本輪穩定，下一輪應考慮：

1. 把 `displayState` 內未使用的 derived/store export 清理掉
2. 評估是否將 displayState 逐步改寫為更 Svelte 5 原生的 state module 形式
3. 繼續縮小 `function/runtime/*` 對 legacy shell 的依賴
