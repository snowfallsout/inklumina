# DEVNOTES_2026-05-02T200000Z

## 背景

在刪除 `function/session/*` 與 `function/runtime/index.ts` 之後，`function/runtime` 還殘留一個明確的公開面：

- `src/lib/function/runtime/types.ts`

這個檔案目前只承擔 runtime 型別定義，不承擔任何行為；同時 `src/lib/services/display/types.ts` 已被視為 display runtime / service 的正式 type entrypoint，但它仍只是對 `function/runtime/types.ts` 的薄 re-export。

這會造成兩個問題：

1. `function/runtime` 看起來仍然是外部 type owner
2. runtime 內部與外部仍共同依賴舊路徑，讓 `services/display/types.ts` 無法成為真正 canonical owner

## 本輪決策

### A. 將 runtime type owner 正式移入 `services/display/types.ts`

本輪把 runtime 型別定義直接放入：

- `src/lib/services/display/types.ts`

這讓 display service 層正式接手 runtime 的 public type surface。

### B. `function/runtime/*` 內部改吃 canonical type entrypoint

本輪同步把 runtime 內部檔案中的 `./types` import 改成：

- `$lib/services/display/types`

這樣 `function/runtime` 不再持有自己的獨立 type owner。

### C. 刪除舊 `function/runtime/types.ts`

完成改線後，刪除 `src/lib/function/runtime/types.ts`。

## 本輪不做

本輪刻意不做以下事項：

- 不移動 runtime 行為實作到 `services/display/*`
- 不處理 `function/runtime/legacy.ts` 的 global bridge 邏輯
- 不改 `function/runtime/camera.ts` 的初始化流程
- 不改 `shared/contracts.ts`

## 驗證方式

本輪完成後需確認：

- `npm run check` 通過
- workspace 中不再有 `function/runtime/types` import
- `services/display/types.ts` 成為正式 runtime type owner

## 下一步

若本輪穩定，下一步應優先盤點：

1. `function/runtime/legacy.ts` 是否可拆成 `services/display` bridge 與真正必要的 global type
2. `function/runtime/dom.ts` 是否可再縮小成純初始化 glue
3. `function/runtime/camera.ts` 是否應拆分 camera boot 與 mediapipe loader
