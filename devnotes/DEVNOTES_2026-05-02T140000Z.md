# DEVNOTES_2026-05-02T140000Z

## 背景

在 mobile flow 收斂到 Svelte 5 基線之後，下一個阻塞點不再是 component 語法，而是跨端 socket 契約仍然分散：

- mobile page 直接對 `submit_mbti`、`lucky_color` 自行定義局部型別。
- display service 與 runtime 各自處理 `state`、`spawn_particles`、`session_reset`，但 payload 仍大量依賴 `any`。
- `src/lib/utils/state.ts`、`src/lib/function/runtime/realtime.ts`、`src/lib/function/session/client.ts` 已經假設存在 `$lib/shared/contracts`，但專案內實際上沒有這個檔案。
- dev server 與正式 server 的 socket emit 結構也沒有被一份正式型別綁定，因此很容易在欄位上漂移。

這代表現在最需要的不是再補單點型別，而是把 socket 與 session response 的資料契約抽成正式 shared contract，讓 mobile、display、server 對的是同一份來源。

## 本次目標

這一輪 shared contract 收斂要完成四件事：

1. 建立真正存在的 `src/lib/shared/contracts.ts`，集中定義事件 payload 與 session API response 型別。
2. 建立 `src/lib/shared/socket-client.ts`，讓 client 端 socket 建立與事件註冊走同一組泛型。
3. 讓 `src/lib/services/socket.ts` 不再以字串 + `any` 作為主要介面，而是使用 shared contract。
4. 讓 `src/lib/server/socket.server.ts` 與 `src/lib/server/vite-socket-plugin.ts` emit / on 的事件 payload 對齊同一份 contract。

## 設計原則

### 1. 事件名稱不新增抽象層，只 formalize 現有名稱

本次不重新命名事件；沿用目前已存在並對照 `static/server.js` 的名稱：

- `state`
- `submit_mbti`
- `spawn_particles`
- `session_reset`
- `lucky_color`
- `mbti:init`

這樣能把重構成本壓到最低，同時先把「名字一樣但欄位可能不同」的風險降下來。

### 2. 先收斂 payload，再收斂 ownership

shared contract 的目的不是把所有 runtime 問題一次解完，而是先讓這些跨端資料流有可檢查的邊界：

- 誰送什麼
- 誰收什麼
- 哪些欄位可選
- 哪些欄位是必需

等 contract 穩定之後，再回頭處理 display state ownership 或更細的 runtime adapter 切分，成本才會下降。

### 3. session REST response 也納入同一層

除了 socket payload，display session panel 與 session client 也已經依賴 shared contract 型別。因此這一輪一起 formalize：

- `SessionRecord`
- `SessionSummary`
- `SessionsOverviewResponse`
- `SessionMutationResponse`
- `DeleteSessionResponse`

這樣 `session` REST API 與 `socket` event 都會落在同一份共享資料定義下，不再分裂成兩套邊界。

## 預期影響

完成後，專案應達成以下狀態：

- `src/lib/utils/state.ts` 與 `src/lib/function/runtime/realtime.ts` 的 import 不再指向不存在檔案。
- mobile / display / server 對同一事件名稱使用同一 payload 形狀。
- 後續若要修改 `submit_mbti` 或 `spawn_particles` 的欄位，編譯器會同時提示三端受影響位置。
- shared contract 會成為後續 display state ownership 與 session API cleanup 的正式邊界。

## 驗證方式

本輪完成後應至少執行：

- `npm run check`

若 typecheck 通過，表示 shared contract 已經至少在型別層成為可用的單一來源。
