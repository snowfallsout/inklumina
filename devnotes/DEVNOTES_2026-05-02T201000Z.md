# DEVNOTES_2026-05-02T201000Z

## 背景

在 `function/runtime/types.ts` 刪除之後，`function/runtime` 還有一個明確的 owner 殘留：

- `src/lib/function/runtime/legacy.ts`

這個檔案負責：

- legacy window bridge type
- 舊 DOM legend / session label 的同步
- display runtime 啟動前後的 global bridge 註冊

但它本質上不是 runtime algorithm 本身，而是 display service 與 legacy 全域橋接邏輯。把它放在 `function/runtime` 會讓 runtime 內核與外部 bridge 職責混在一起。

## 本輪決策

### A. 將 legacy bridge owner 移入 `services/display/legacy.ts`

本輪新增：

- `src/lib/services/display/legacy.ts`

用它承接原本 `function/runtime/legacy.ts` 的 bridge 行為與 type。

### B. 保持 runtime 內部透過 service owner 存取 legacy bridge

本輪更新：

- `function/runtime/camera.ts`
- `function/runtime/realtime.ts`

讓 runtime 內部改由 `services/display/legacy.ts` 取得 bridge type / helper。

### C. 刪除舊 `function/runtime/legacy.ts`

完成改線後，刪除舊檔，避免 `function/runtime` 繼續持有 bridge owner。

## 本輪不做

本輪刻意不做以下事項：

- 不改 `function/runtime/realtime.ts` 的 socket 行為
- 不改 `function/runtime/camera.ts` 的 loader/fallback 行為
- 不改 `services/display/runtime.ts` 的 public API

## 驗證方式

本輪完成後需確認：

- `npm run check` 通過
- workspace 中不再有 `function/runtime/legacy` import
- legacy bridge 的正式 owner 變成 `services/display/legacy.ts`

## 下一步

若本輪穩定，下一步應優先盤點：

1. `function/runtime/realtime.ts` 是否也應搬到 `services/display/*`
2. `function/runtime/dom.ts` 是否只剩啟動 glue，可再往 service 層上收
3. `function/runtime/camera.ts` 是否可抽出更小的 camera loader/service
