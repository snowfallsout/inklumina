# DEVNOTES_2026-05-02T150000Z

## 背景

目前專案已經有明確的 Svelte 5 分層：

- `state`：共享 reactive state
- `services`：browser adapter / network / client orchestration
- `server`：server-side contract 與 socket / session API
- `constants` / `config`：靜態設定
- `core`：粒子引擎與 sprite cache 這類較純的演算法核心

但 `src/lib/function/**` 仍保留舊時代的混合結構：

- `function/runtime` 內同時有 browser runtime bridge、DOM wiring、gesture state machine、camera setup、socket consume
- `function/session` 內同時有 session REST client、QR 生成、panel DOM interaction
- 外部公開邊界仍透過 `function/*` 或 `utils/session.ts` 對 `function/session/*` 的 re-export 暴露

這讓架構雖然能跑，但 ownership 不清楚：Svelte 5 專案表面上已經有 `state/services/server/core`，實際上卻還有一套平行的 `function/runtime/session`。

## Svelte 5 架構判準

依 Svelte 5 與本專案目前的方向，新的判準應該是：

1. `.svelte.ts` 只持有共享 reactive state，不擔任大型 imperative runtime 的唯一 owner。
2. browser-only orchestration 應進入 `services`，讓 component / runtime 只呼叫清楚的 API。
3. 純 helper 應進入 `utils`，不再混在 runtime 與 panel 內。
4. 粒子 engine、sprite cache、draw math 這類 imperative 但可隔離的邏輯，可保留在 `core` 或 runtime internals。
5. `function/` 若暫時存在，只能作 internal implementation detail 或 compatibility layer，不能再作為新公共結構。

## 重構決策

### A. `function/session` 應優先退出 public surface

原因：

- 這條線本質是 client-side orchestration，不是 engine。
- 現有外部入口只剩 `utils/session.ts` 對 `function/session/*` 的 re-export。
- 這是最便宜、最不破壞行為、也最符合 Svelte 5 `services + utils` 分層的第一刀。

調整方向：

- `utils/session.ts` 只保留純 helper：`STORAGE_KEY`、`sanitizeHost`、`buildJoinUrl`
- 新增 `services/displaySession.ts`，承接：
  - session panel DOM 操作
  - session REST client orchestration
  - join QR / corner QR client 行為
- display runtime 改從 `services/displaySession.ts` 取用這些能力

### B. `function/runtime` 先退回 internal runtime，而非立即整包搬家

原因：

- runtime 內仍有大量 imperative canvas / MediaPipe / gesture engine 細節。
- 這些不是不能存在，而是要被清楚標示為 internal runtime implementation。
- 若在沒有先切清 public API 的情況下整包移動，只會得到大規模 rename，而不是實際的 ownership 改善。

第一階段方向：

- 新增 `services/displayRuntime.ts` 作為 flatter public entry
- `function/runtime/*` 暫留為 internal implementation
- 後續再依責任拆成：
  - `core/display/*`：engine / particle / sprite / draw math
  - `services/displayRuntime.ts`：browser bootstrap / glue

## 本輪實作範圍

這一輪只做第一階段：

1. 新增本文件作為正式設計說明。
2. 將 `function/session` 的 public surface 抽到 `services/displaySession.ts`。
3. 讓 `utils/session.ts` 回到純 helper 檔。
4. 新增 `services/displayRuntime.ts` 作為未來外部入口。
5. 把現有 runtime import 改用新的 flatter public path。

## 明確不做

本輪不做以下事情：

- 不刪除整個 `function/runtime` 目錄
- 不重新分拆所有粒子與 camera engine 到新資料夾
- 不修改 protected `static/**`
- 不在同一輪裡處理所有 legacy runtime DOM ownership

## 驗證方式

本輪完成後至少要確認：

- `npm run check` 通過
- `utils/session.ts` 不再是 `function/session/*` 的 re-export barrel
- display runtime 對 session orchestration 的 import 已轉到 `services/displaySession.ts`

2. 把更純的 engine pieces 往 `core/display/*` 收斂
3. 最後把 `function/` 留下的 compatibility shell 逐步移除
